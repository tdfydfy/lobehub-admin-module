## 1. 当前定位

`lobehub-admin-module` 是 `lobehub` 的独立外挂管理模块。

它不修改主应用源码，直接复用现有 PostgreSQL 数据，负责：
- 项目管理
- 项目成员管理
- 项目模板助手管理
- 项目级批量助手配置与刷新
- 项目经营日报
- 项目级对话统计与下钻查看

## 3. 当前关键行为

### 3.1 全局自动下发已关闭

当前环境已切换到项目级控制模式：
- `public.system_provisioning_config.enabled = false`
- `public.users` 上的 `trg_provision_on_user_insert` 已禁用

因此：
- 新用户注册后保持空白账号
- 只有通过项目管理页面手动配置后，成员才会获得项目助手

### 3.2 项目托管助手的行为

一个项目对同一个成员只维护一条“项目托管助手”记录。

这意味着：
- 第一次配置时会创建项目托管助手
- 后续刷新或更换模板时，会更新这条项目托管助手
- 不会因为换模板而不断累积多条项目助手副本

补充规则：
- 官方助手的唯一身份只看 `(project_id, user_id)` 对应的 `project_managed_agents`
- 官方助手不按标题识别
- 用户私人助手即使与模板助手同名，也不作为官方助手识别依据
- 运行时优先复用映射中的官方助手/官方会话；如果历史映射脏了，优先用增量修复脚本处理，而不是继续扩展运行时分支

### 3.3 用户自建助手不会被删除

当前逻辑不会删除成员自己创建的其他助手。

边界如下：
- 项目托管助手：会被更新
- 用户自己手动创建的其他助手：保留
- 系统内置助手：保留

### 3.4 技能复制仍有覆盖风险

当前 skill 复制策略仍是按 `(user_id, name)` 幂等更新。

这意味着：
- 如果成员自己已有同名 skill
- 且项目模板里也有同名 skill
- 刷新时该同名 skill 可能被模板版本覆盖

这是当前最重要的未收敛风险之一。

### 3.5 数据查看权限边界已明确

当前“数据查看”能力按操作者角色划分：
- 系统管理员：可直接查看 `crm` / `lobehub_admin` / `public`
- 项目管理员：仅可查看 `crm` schema
- 项目管理员查看 `crm` 表时，后端会强制追加 `project = 自己所在项目名称`
- 如果 `crm` 表不存在 `project` 字段，则返回空数据而不是报错

这是为后续 CRM 正式落表预留的访问规则。

### 3.6 前端时间显示默认东八区

当前前端展示层会将标准 UTC 时间字符串转换为东八区显示。

适用范围：
- 首页项目详情页
- 项目成员运营报表
- 项目对话统计
- 数据查看页

数据库原始存储值不做改写。

### 3.7 对话统计口径已固定

当前“对话统计”能力的默认统计口径是：
- 只统计项目托管会话里的 `topic`
- 不统计用户私人会话或其他非项目会话
- 时间范围按东八区自然日切分

下钻路径为：
- 成员列表 -> topic 清单
- topic 清单 -> 消息详情

这保证了统计结果与项目托管助手的实际使用情况一致。

### 3.8 日报口径与调度行为已固定

当前“项目经营日报”能力的默认行为是：
- 统计窗口按项目时区与营业日截点切分，而不是按服务端自然日
- 数据源只取项目托管助手会话内、落在营业日窗口内的真实消息
- 视角固定为项目经营管理，不评价单个销售表现
- 提示词结构固定为两层：系统提示词 + 项目补充要求
- 模型解析顺序为：项目覆盖 -> 环境默认 -> `VOLCENGINE_API_KEY` 存在时默认 `volcengine` -> `fallback`
- 服务进程启动时会恢复未完成日报任务，并每 60 秒扫描一次已启用项目的到点日报
- LLM 生成失败时会回退到内建规则摘要，不会直接丢失整份日报

## 4. 当前已经处理过的问题

### 4.1 CORS 问题

已修复页面保存模板时报 `Failed to fetch` 的问题。

当前后端已同时允许：
- `http://127.0.0.1:4173`
- `http://localhost:4173`

### 4.2 模板切换时 `agents_pkey` 冲突

已修复“更换新模板后刷新成员助手时报 `duplicate key value violates unique constraint "agents_pkey"`”的问题。

修复思路：
- 项目托管助手按“项目 + 用户”稳定复用
- 换模板时更新原托管助手，而不是再插新主键

### 4.3 官方助手跳过条件过宽

已修复“官方助手存在但托管会话缺失时，普通配置误判为已完成并跳过”的问题。

当前规则：
- 只有 `managed_agent_id` 和 `managed_session_id` 都存在，才允许直接 `skipped`
- 少一个都要补齐

### 4.4 官方助手身份收敛

已将官方助手识别规则收敛为：
- 每个 `(project_id, user_id)` 最多 1 个官方助手
- 每个 `(project_id, user_id)` 最多 1 个官方会话
- 官方身份只看 `project_managed_agents`
- 不再按助手标题推断官方身份

### 4.5 模板助手候选列表过脏

已过滤掉不适合作为模板的内部助手：
- `inbox`
- `page-agent`
- `agent-builder`
- `group-agent-builder`

当前模板候选列表只保留：
- 有明确标题的助手
- 且不是这些内部 slug

### 4.6 批量任务并发与成员快照

已补上两层保护：
- 同一项目已有 `pending/running` 任务时，后端拒绝再次创建新任务
- 前端在任务运行中会禁用“为成员配置助手 / 刷新成员助手”按钮

同时当前任务会在创建时固化成员快照：
- 任务创建时先写入 `provision_job_items`
- 后台执行时优先消费这份快照
- 不再按执行时的当前成员列表重新扫描

这避免了：
- 同项目并发批量任务互相踩写
- 任务排队/恢复后执行对象与点击时对象不一致

### 4.7 Topic 统计已改为用户可见消息口径

已收紧 Topic 统计页展示口径：
- Topic 列表“消息数”只统计用户可见消息
- Topic 预览只取最后一条可见消息
- Topic 详情只返回用户可见消息

当前第一版“可见消息”定义为：
- `role in ('user', 'assistant')`
- 且 `content` 非空

因此：
- `tool`
- `system`
- 空白中间帧
- 原始 `editor_data`

不再在 Topic 统计页默认展示

## 7. 当前建议使用方式

在当前阶段，推荐只按以下路径使用：

1. 先选定一个项目
2. 先保存模板配置
3. 再执行“为成员配置助手”或“刷新成员助手”
4. 如需使用日报，先确认日报设置里的时区、营业日截点、项目补充要求与模型配置
5. 若换模板，直接刷新成员助手，不需要手动删除原项目助手

这是当前最稳定的使用路径。
