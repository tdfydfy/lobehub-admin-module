# 管理模块架构

## 1. 定位

该模块是 `lobehub` 的外挂管理系统，不与主应用代码混合部署。

模块当前主要做三件事：
- 管理项目与项目成员
- 为项目成员批量配置模板助手
- 提供项目报表、对话统计与原始数据查看入口

业务数据仍保留在原有 `lobehub` 数据库中。

## 2. 总体结构

建议分为四层：

### 2.1 数据复制内核

职责：
- 从模板用户复制技能到目标用户
- 从模板助手复制助手配置到目标用户
- 为目标用户创建项目托管会话
- 记录项目托管助手与会话映射

说明：
- 该层直接复用并演进现有 `scripts/install-universal-auto-provision.sql` 的能力
- 新设计不再依赖系统级模板表
- 所有输入都显式传参

### 2.2 项目域模型层

职责：
- 管理项目
- 管理项目成员
- 保存项目模板
- 跟踪项目级批量任务

核心对象：
- `projects`
- `project_members`
- `project_templates`
- `project_managed_agents`
- `provision_jobs`
- `provision_job_items`
- `system_admins`

为避免污染主表空间，统一放在 `lobehub_admin` schema 中。

### 2.3 管理 API 层

职责：
- 系统管理员创建项目
- 项目管理员添加成员
- 项目管理员设置模板
- 发起批量配置与刷新
- 查询任务进度
- 查询报表数据
- 查询项目对话统计与对话详情
- 查询原始表数据

该层只做编排与权限校验，不重复实现复制逻辑。

### 2.4 管理前端层

页面建议保持最小结构：
- `/projects`
- `/projects/[id]`

项目详情页包含 5 个 tab：
- 成员管理
- 助手配置
- 对话统计
- 数据查看
- 数据报表

## 3. 与主应用的边界

### 3.1 不做的事情

- 不改动 `lobehub` 原始前端页面
- 不把管理逻辑塞进主项目路由
- 不依赖主应用内部组件或私有状态

### 3.2 直接复用的事情

- `public.users`
- `public.agents`
- `public.agent_skills`
- `public.sessions`
- `public.agents_to_sessions`
- `public.user_settings`

## 4. 项目级配置原则

### 4.1 用户注册后保持空白

新的项目管理方案要求：
- 用户注册后不自动发助手
- 只有项目管理员显式批量配置后才获得项目助手

因此旧的 `system_provisioning_config` 自动下发应默认关闭。

### 4.2 项目模板只有一个

MVP 阶段每个项目只允许一个模板：
- 一个模板用户
- 一个模板助手

后续如需多模板，再在项目层扩展，不在第一版引入复杂度。

### 4.3 不强制改写默认助手

用户可能属于多个项目，也可能已有私人助手。

因此第一版默认行为是：
- 复制项目助手
- 建立项目托管会话
- 不主动改写 `user_settings.default_agent`

## 5. 冲突处理原则

### 5.1 托管助手冲突

项目助手不能和用户私人助手混为一谈。

解决方式：
- 为项目托管助手创建独立 slug
- 用 `project_managed_agents` 映射项目、用户、助手、会话

### 5.2 技能复制冲突

LobeHub 的 skill 是用户级对象，不是项目级对象。

风险：
- 复制模板用户所有 skill 时，可能覆盖目标用户同名 skill

第一版处理：
- 保留 `copy_skills` 开关
- 默认允许复制
- 在 UI 和操作说明中明确风险

## 6. 批量任务模型

批量配置不能设计为纯同步接口。

第一版数据库会先提供任务表和同步执行函数，后续独立管理端可演进为：
- API 创建任务
- 后台 worker 执行
- 前端轮询进度

任务状态建议：
- `pending`
- `running`
- `completed`
- `partial`
- `failed`

明细状态建议：
- `pending`
- `running`
- `success`
- `failed`
- `skipped`

## 7. 数据能力边界

当前数据能力分为三类：

### 7.1 项目成员运营报表

当前已实现基于现有 `lobehub_admin` / `public` 表的真实查询：
- 报表按项目成员范围过滤
- 支持用户过滤
- 支持时间过滤
- 支持分页和导出

这部分用于管理项目成员、托管助手、任务结果等现有业务数据。

### 7.2 项目对话统计

当前已实现基于 `public.topics` / `public.messages` / `lobehub_admin.project_managed_agents` 的项目级对话统计：
- 统计口径限定为“项目托管会话”内创建的 `topic`
- 支持当日、近三天、近七天、一个月、指定日期范围
- 时间边界按东八区自然日计算
- 支持从成员下钻到 topic 清单
- 支持从 topic 清单下钻到消息详情

这部分用于查看项目成员在项目托管助手上的实际对话使用情况，不混入用户私人会话。

### 7.3 数据查看

当前已实现通用原始表查看能力，作为 CRM 正式落表前的预留入口：
- 系统管理员可查看 `crm` / `lobehub_admin` / `public`
- 项目管理员仅可查看 `crm`
- 项目管理员读取 `crm` 表时，后端强制按 `project = 自己所在项目名称` 过滤
- 如果表中不存在 `project` 字段，则项目管理员视角返回空数据

该能力用于直接查看原始记录，不负责业务语义汇总。

### 7.4 仍未完成的部分

虽然已具备“项目成员运营报表”“项目对话统计”和“数据查看”，但真正面向 CRM 业务语义的专用报表仍未完成：
- 尚未沉淀 CRM 业务指标定义
- 尚未实现 CRM 业务图表或聚合统计
- 尚未引入正式的 CRM 业务查询模型
