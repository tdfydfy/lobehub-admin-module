# 知识域架构

## 目标

将知识内容从零散的知识型 skill 中逐步抽离，统一收口到两个知识域：

- 全局知识域
- 项目知识域

动作型能力仍然保留为工具或 skill。

## 两个知识域

### 全局知识域

用途：

- 面向所有项目、所有成员共享
- 由系统管理员统一维护

典型内容：

- 客户判断标准
- 追问策略
- 销售 SOP
- 异议处理原则
- 学校 / 学区合规口径
- 价格表达原则
- 竞品对比原则

权限：

- 系统管理员：增删改查
- 项目管理员：只读
- 普通成员：仅通过 agent 问答间接使用

### 项目知识域

用途：

- 面向单个项目
- 由项目管理员维护

典型内容：

- 项目介绍
- 区位信息
- 户型资料
- 项目 FAQ
- 项目价格口径
- 项目竞品资料
- 项目特殊要求

权限：

- 系统管理员：增删改查
- 项目管理员：增删改查
- 普通成员：仅通过 agent 问答间接使用

## 知识地图文档

每个知识域都应存在一个 `00-map` 文档：

- 全局知识域：`00-global-knowledge-map`
- 项目知识域：`00-project-knowledge-map`

作用：

- 给管理员提供知识总览
- 给系统提供轻量级路由参考
- 明确知识分类和优先级

知识地图文档不是硬编码路由表，不需要穷举“每个问题读哪篇文档”，只需要说明：

- 当前知识域有哪些大类
- 每类大概有哪些文档
- 什么问题优先看哪类
- 项目知识和全局知识如何配合

## 推荐文档结构

每篇知识文档保持轻量：

- `title`
- `description`
- `content_md`
- `status`
- `is_entry`
- `sort_order`

后续如有需要，可再补：

- `category`
- `keywords`

不建议一开始引入过重的 tags / taxonomy 体系。

## 统一知识工具

agent 层面尽量只暴露一个统一知识工具：

- `queryKnowledge(question)`

后端负责：

1. 判断知识域
2. 参考知识地图缩小范围
3. 选择 1-3 篇候选文档
4. 返回给模型

## 路由原则

### 项目助手

- 优先项目知识
- 再补全局知识

### 非项目助手

- 默认只用全局知识

### 文档选择

- 先按知识地图收敛大类
- 再按文档标题 / 描述 / 正文轻量匹配
- 搜索是兜底，不是默认入口

## 和 skill 的关系

保留：

- 真正执行动作的 skill / tool
- 例如 CRM 查询、CRM 更新、外部 API 调用

迁移：

- 纯知识型 skill
- 纯规则型 skill
- 纯话术型 skill

这些内容应逐步迁到全局知识域或项目知识域。

## 开发顺序建议

1. 先完成项目知识域
2. 再补全局知识域
3. 补齐两个 `00-map`
4. 统一 `queryKnowledge`
5. 逐步清理旧的知识型 skill
