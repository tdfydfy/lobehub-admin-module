# Metrics V1 口径说明

更新时间：2026-03-30

状态：待确认后实施

适用范围：`lobehub-admin-module` 多项目经营看板、项目概览、日报 V2、移动端总览

## 1. 目标

本文件用于固化本轮升级的统一经营口径，避免以下问题持续存在：

- topic 统计按 `topic.created_at` 算，日报按 `message.created_at` 算
- 同一个老 `topic` 今天新增消息时，日报和概览数字不一致
- “来访 / 首访 / 复访 / 活跃”在不同页面语义不同

本文件是 V1 的唯一统计口径来源。

## 2. V1 适用前提

### 2.1 客户识别前提

- V1 正式采用 `topic = customer`
- 运营规则继续保持：`一个 topic 只服务一个客户`
- 如后续发现一个客户会拆成多个 `topic`，则需进入 V2 的 CRM 客户实体化升级

### 2.2 数据范围前提

V1 只统计以下范围：

- 项目成员
- 项目托管会话
- 托管会话内创建的 `topic`
- 落在指定营业日窗口内的有效消息

不统计：

- 用户私人会话
- 非项目托管会话
- `tool` 消息
- 纯 CRM 字段补录但未进入消息流的操作

### 2.3 营业日切分前提

- 所有指标按“项目时区 + 营业日截点”切分
- 不按服务器自然日切分
- 不按固定东八区自然日兜底
- 同一项目的概览、日报、移动端必须使用同一营业日窗口

## 3. 基础对象定义

### 3.1 有效消息

V1 有效消息定义：

- 非 `tool`
- 且存在用户可见内容

可见内容来源优先级：

1. `content`
2. `summary`
3. `editor_data`

### 3.2 来访消息

V1 的“来访”只基于有效 `user` 消息判断。

说明：

- assistant 总结、assistant 回答、本地摘要更新，不单独构成来访
- 只有客户侧新输入进入消息流，才算来访或复访

### 3.3 活跃 topic

V1 的活跃 topic 定义为：

- 在营业日窗口内至少拥有 1 条有效消息的 `topic`

### 3.4 来访客户

V1 的来访客户定义为：

- 在营业日窗口内至少拥有 1 条有效 `user` 消息的 `topic`

## 4. 核心指标定义

| 指标 | 层级 | 定义 | 说明 |
| --- | --- | --- | --- |
| `newTopicCount` | 项目 / 成员 | 营业日窗口内新建的 `topic` 数 | 偏“新增对话” |
| `activeTopicCount` | 项目 / 成员 | 营业日窗口内存在有效消息的 `topic` 数 | 老 `topic` 今日更新也算 |
| `visitCustomerCount` | 项目 / 成员 | 营业日窗口内存在有效 `user` 消息的 `topic` 数 | 经营口径中的来访 |
| `firstVisitCount` | 项目 / 成员 | 当日来访 topic 中，第一条有效 `user` 消息业务日就是今天的数量 | 首访 |
| `revisitCount` | 项目 / 成员 | 当日来访 topic 中，第一条有效 `user` 消息业务日早于今天的数量 | 复访 |
| `activeMemberCount` | 项目 | 当日至少拥有 1 个活跃 topic 的成员数 | 成员活跃度 |
| `visibleMessageCount` | 项目 / topic | 营业日窗口内所有有效消息数 | 非 `tool`、有可见内容 |
| `userMessageCount` | 项目 / topic | 营业日窗口内所有有效 `user` 消息数 | 来访基础 |
| `assistantMessageCount` | 项目 / topic | 营业日窗口内所有有效 `assistant` 消息数 | 辅助分析 |
| `aIntentCount` | 项目 | 当日最新有效等级为 `A` 的 topic 数 | 高意向 |
| `bIntentCount` | 项目 | 当日最新有效等级为 `B` 的 topic 数 | 高意向 |
| `cIntentCount` | 项目 | 当日最新有效等级为 `C` 的 topic 数 | 中意向 |
| `dIntentCount` | 项目 | 当日最新有效等级为 `D` 的 topic 数 | 低意向 |
| `missingIntentCount` | 项目 | 当日无有效等级的 topic 数 | 待补信息 |

## 5. 首访与复访推导规则

### 5.1 首次来访时间

一个 `topic` 的首次来访时间定义为：

- 该 `topic` 第一条有效 `user` 消息时间

### 5.2 当日是否来访

一个 `topic` 当日来访的判定条件：

- 在本营业日窗口内存在至少 1 条有效 `user` 消息

### 5.3 当日是否首访

一个 `topic` 当日首访的判定条件：

- 当日来访 = true
- 且第一条有效 `user` 消息所属业务日 = 当天业务日

### 5.4 当日是否复访

一个 `topic` 当日复访的判定条件：

- 当日来访 = true
- 且第一条有效 `user` 消息所属业务日 < 当天业务日

### 5.5 老 topic 今日更新的处理

老 `topic` 今天新增客户消息时，应当体现为：

- 不算新增对话
- 算活跃对话
- 算来访客户
- 算复访客户

## 6. 意向等级口径

### 6.1 唯一来源

意向等级唯一来源：

- `lobehub` 对话中 AI 已输出并落库的有效等级结果

### 6.2 当日取值规则

同一 `topic` 在同一营业日内出现多次等级结果时：

- 默认取当日最新一条有效等级

如同一营业日内等级发生变化：

- 以最新有效结果为准

### 6.3 无等级处理

如某 `topic` 当日没有有效等级：

- 不归入低意向
- 记为 `missingIntentCount`
- 日报中进入“待补信息”

## 7. 指标层、分析层、明细层口径

### 7.1 指标层

指标层只读取结构化日事实，不直接拼原始消息。

适用页面：

- 组合看板
- 单项目概览
- 移动端总览
- 日报顶部数字

### 7.2 分析层

分析层读取营业日窗口内的有效 `user + assistant` 消息，用于生成：

- 日报正文摘要
- 本次新增信息摘要
- 管理问题与动作

分析层默认不读取整条 topic 全历史，避免噪音和上下文膨胀。

### 7.3 明细层

明细层继续展示该 `topic` 的全量可见消息。

适用页面：

- topic 详情
- 从日报下钻进入的客户明细

## 8. 典型样例

### 样例 1：新 topic，今天第一次有客户消息

预期：

- `newTopicCount = 1`
- `activeTopicCount = 1`
- `visitCustomerCount = 1`
- `firstVisitCount = 1`
- `revisitCount = 0`

### 样例 2：老 topic，今天新增客户消息

预期：

- `newTopicCount = 0`
- `activeTopicCount = 1`
- `visitCustomerCount = 1`
- `firstVisitCount = 0`
- `revisitCount = 1`

### 样例 3：老 topic，今天只有 assistant 总结，没有客户消息

预期：

- `newTopicCount = 0`
- `activeTopicCount = 1`
- `visitCustomerCount = 0`
- `firstVisitCount = 0`
- `revisitCount = 0`

### 样例 4：只有 CRM 字段补录，没有消息流变更

预期：

- 不计入来访
- 不计入首访
- 不计入复访
- 不计入新增或活跃对话

## 9. 页面映射规则

### 9.1 组合看板

组合看板必须展示：

- 项目数
- 今日来访
- 今日首访
- 今日复访
- 今日新增对话
- 今日活跃对话
- 今日活跃成员
- 今日 A/B
- 今日待补信息

### 9.2 单项目概览

单项目概览必须和组合看板项目行一致。

### 9.3 日报顶部数字

日报顶部数字必须和单项目概览一致。

### 9.4 日报客户标签

日报客户条目需支持：

- `首访`
- `复访`

### 9.5 移动端

移动端可简化展示，但数字不能另起口径。

## 10. 明确排除项

V1 明确不支持以下口径：

- 纯 CRM 字段更新触发的复访
- topic 外部行为事件触发的复访
- 多 topic 合并识别同一客户
- 一客户多次切 topic 的自动归并

## 11. 何时进入 V2

若灰度中出现以下情况，则应进入 V2，而不是继续在 V1 口径上打补丁：

- 大量客户会跨多个 topic
- CRM 补录必须被统计为复访
- 需要跨 topic 聚合同一客户生命周期
- 需要正式客户池、客户状态流转、客户归因

V2 的方向应为：

- 正式 `crm_customers`
- `customer_topic_links`
- `customer_interaction_events`

## 12. 最终结论

V1 的核心不是把系统做成 CRM，而是先把经营口径做稳。

只要以下三点成立，V1 就能支撑本轮多项目升级：

- `topic = customer`
- 来访只认有效 `user` 消息
- 组合看板、项目概览、日报顶部数字共用同一事实来源
