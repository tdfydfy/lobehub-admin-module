# 日报意向等级与重点客户展示升级方案

## 1. 背景

当前日报已经能基于项目托管会话生成经营摘要，但两点不够贴业务：

1. 高意向客户的识别目前仍偏日报侧归纳，和 `lobehub` 对话里已经产出的 AI 客户判断结果没有完全打通。
2. 重点客户展示信息偏粗，缺少最关键的管理信息，但如果把所有原始记录都堆上去，又会明显增加复杂度。

基于当前业务口径，后续日报应做两件事：

- 复用 `lobehub` 对话里已有的 `A/B/C/D` 客户意向判断，不在日报侧重复造一套标准。
- 重点客户展示做减法，只保留最重要的信息和跳转入口。

## 2. 目标

- 让日报中的“高意向客户”口径稳定、可解释。
- 让管理端和移动端都能快速看到今日重点客户，而不是翻很多原始内容。
- 对没有输出意向等级的客户，明确标识为“信息不足”，形成补录提醒。
- 不引入复杂冗余字段，不把日报做成客户管理系统。

## 3. 范围

本次只覆盖两项：

- `A/B/C/D` 意向等级接入日报
- 重点客户的简洁展示

本次不覆盖：

- CRM 客户实体改造
- 日报侧重新训练或重新推断意向等级
- 大而全的客户字段表格
- 周报/月报扩展
- 销售录入流程改造

## 4. 核心口径

- 意向等级唯一来源：`lobehub` 对话中 AI 已输出并写入日志的结果。
- 运营与使用规范上约束：`一个 topic 只服务一个客户`。
- 日报不再根据关键词和消息量自行判定高意向。
- `A/B` 归类为高意向客户。
- `C/D` 不进入默认重点客户列表。
- 没有意向等级时，不归为低意向，而标记为“信息不足”。
- 同一客户在同一营业日有多条分析结果时，默认取“最新一条有效等级”。
- 如同一营业日内等级发生变化，以最新有效结果为准。

## 5. 展示原则

日报主界面只保留 5 个核心信息：

- 客户标识
- 销售员
- 意向等级
- 一句话结论
- 查看对话入口

以下内容不放到主列表：

- 长篇初始描述
- 原始聊天全文
- 大量补充字段
- 多层嵌套分析结果

初始描述、证据消息、原始分析过程，只放到二级详情中查看。

移动端展示补充原则：

- 数字卡片优先横向排列，标签和值同一行，避免竖向堆叠占空间。
- 日报详情顶部的数字卡片固定为 4 个并尽量同排展示：
  - 来访组数
  - A/B 类高意向
  - C/D 类中低意向
  - 待补信息
- 四个数字块必须来自同一套基础统计，保证：
  - 高意向 = A + B
  - 总来访 = A + B + C + D + 待补信息
- 顶部已有营业日与窗口信息时，数字卡片不再重复展示日期。
- 重点客户卡片中，意向等级使用突出色标签展示。
- 意向等级标签放在卡片右上角，销售员姓名放在客户标题下方，不放在卡片右侧挤压布局。
- 重点客户标题默认单行展示，不允许因右上角标签挤压导致标题换行。
- “管理关注点”和“建议动作”在移动端合并展示为“问题-动作”卡片，减少重复信息。
- “问题”必须是具体问题，不使用“某某问题是管理层需要统一解决的问题”这类空话；应直接写出例如“可售房源口径不足”“预算与价格明显错配”等具体问题。
- 优先级和高低判断统一用中文展示，例如“高优先级 / 中优先级 / 低优先级”。
- 与其用长句重复描述“有几组客户关注这个问题”，不如直接用数字标签展示涉及客户组数。
- `Overview` 中的“最值得关注的客户”和“优先动作”应分行列举，不堆在同一段摘要里。

## 6. 执行阶段

### 阶段 0：口径确认

目标：确认上游 AI 结果的真实落点和使用规则。

需要确认：

- `A/B/C/D` 具体写入哪个日志字段
- 一句话结论是否已有可复用文本
- 同一客户的唯一标识是什么
- 同一天多次分析时取值规则
- 无等级时是否存在可用的“原因说明”

产出：

- 字段映射表
- 样例数据
- 取值规则说明

验收：

- 能明确指出日报从哪里读等级
- 能明确指出“信息不足”怎么识别

### 阶段 1：数据结构接入

目标：把上游客户判断转成日报可消费的结构。

新增的日报内部结构建议：

- `intentSummary`
  - `aCount`
  - `bCount`
  - `cCount`
  - `dCount`
  - `missingCount`
- `keyCustomers`
  - 默认仅放 `A/B`
- `missingInfoCustomers`
  - 放没有意向等级但值得关注的客户

规则：

- `A/B` 默认进入重点池
- `C/D` 只统计，不默认前置展示
- 无等级客户进入“信息不足池”
- 如果没有一句话结论，则允许退回简短摘要，但不允许退回重新判级

验收：

- 每份日报都能清楚给出 `A/B/C/D/信息不足` 数量
- 高意向数量完全可追溯

### 阶段 2：日报内容重构

目标：让日报从“泛摘要”变成“经营重点卡片”。

日报结构调整为三段：

- 今日经营概览
  - 今日客户总数
  - `A/B/C/D` 分布
  - 信息不足数量
- 今日重点客户
  - 仅展示 `A/B`
- 今日待补信息客户
  - 展示“信息不足”

内容规则：

- 重点客户数量控制在一个可读范围内
- 一句话结论要面向经营判断，不面向原始过程复述
- 待补信息客户要明确表达“当前无法判级的原因是信息不足”

验收：

- 用户打开日报后，第一屏就知道今天有没有值得重点跟进的 `A/B`
- 同时也知道哪些客户因为信息不足需要补录

### 阶段 3：Web 端方案

目标：Web 端承担管理、复核和下钻职责。

页面结构：

- 顶部概览卡
  - `A/B/C/D` 数量
  - 信息不足数量
- 重点客户列表
  - 客户标识
  - 销售员
  - 等级
  - 一句话结论
  - 查看对话
- 待补信息列表
  - 客户标识
  - 销售员
  - “信息不足”标签
  - 查看对话

交互规则：

- 默认按 `A > B > 信息不足` 排序
- 列表主视图不展示长文本
- 点击后进入详情抽屉或详情页
- 详情里再看：
  - 初始描述
  - AI 输出的结论
  - 原始证据消息
  - 对话详情跳转
- 日报点进客户明细时，默认展示该 `topic` 的全量消息；如果一个 `topic` 中混入多个客户，视为上游使用规范问题，而不是日报展示层问题。

验收：

- 管理人员在 Web 端 30 秒内可完成“看重点客户 + 看待补信息客户”
- 不需要先打开原始 JSON 才能理解结果

### 阶段 4：移动端方案

目标：移动端只保留“快看重点”的能力。

页面结构：

- 顶部摘要卡
  - `A/B` 总数
  - 信息不足数
  - 今日重点客户数
- 重点客户卡片流
  - 客户标识
  - 销售员
  - 等级
  - 一句话结论
- 待补信息卡片流
  - 客户标识
  - 销售员
  - 信息不足提示
- 卡片点击进入简化详情页
  - 查看初始描述
  - 查看对话入口

移动端约束：

- 不做宽表格
- 不做过多筛选
- 不堆长文本
- 一屏优先展示 `A/B`
- 数字摘要区优先采用横向紧凑布局
- 数字摘要区按 4 个固定指标展示：来访组数、A/B 高意向、C/D 中低意向、待补信息
- 重点客户卡片优先展示“右上角等级标签 + 标题下销售员 + 一句话结论”
- 管理问题与建议动作合并展示，不拆成两个重复区块
- 管理问题优先展示具体问题本身，动作作为对应解决手段跟在后面
- 优先级、风险高低等状态统一使用中文文案

验收：

- 在手机上能直接看出今天有没有重点客户
- 不需要多次展开才能找到关键信息

### 阶段 5：灰度验证

目标：先验证口径和可读性，再推广。

灰度方式：

- 先选 1 到 2 个项目试运行
- 保留旧日报视图一段时间用于对照
- 由业务人工抽样核对：
  - `A/B/C/D` 是否和对话日志一致
  - 无等级客户是否确实信息不足
  - 排序是否符合直觉

重点观察：

- `A/B` 命中率是否稳定
- 信息不足占比是否过高
- 用户是否主要点击重点客户而不是原始 JSON
- 管理人员是否反馈页面更清晰

验收：

- 重点客户列表可直接用于日盘客
- 用户不再提出“重点客户很多但看不清”的问题

### 阶段 6：二期可选优化

稳定后再考虑：

- 信息不足原因细分
- `A/B` 内部优先级排序
- 重点客户详情补更多上下文
- 周报/月报沿用同一口径
- 销售补录关键字段闭环

## 7. 风险

- 上游日志里的 `A/B/C/D` 结构如果不稳定，日报接入会反复返工。
- 同一客户多轮分析如果没有稳定主键，容易产生重复或覆盖错误。
- 如果“信息不足”比例太高，说明问题不在日报，而在上游客户分析输入质量。
- 如果 Web 和移动端想承载太多细节，页面会重新变乱。

## 8. 建议决策

1. 先冻结口径：日报只认对话日志里的 `A/B/C/D`。
2. 先做最小展示：客户、销售、等级、结论、入口。
3. 先跑灰度，再决定要不要加更多字段。
